# Generated Context Standard

## Purpose

This document defines the rules for context artifacts generated by Monad.

Generated context is distinct from human-authored context. Generated artifacts are useful for orientation but should not be trusted as accepted doctrine until a human reviews them.

## Core Rule

Generated context must:

```text
identify itself as generated
list its source files
be deterministic from the same repository state
be reviewable in a diff tool
not contain secrets or credentials
not claim to be accepted unless reviewed
```

## Generated vs Human-Authored

| Property | Human-Authored | Generated |
| --- | --- | --- |
| Frontmatter `generated` | `false` or absent | `true` |
| Frontmatter `reviewed` | not required | `false` until reviewed |
| Trust level | accepted | reviewable |
| Location | `docs/` | `.monad/context/` or `.monad/context/generated/` |
| In git | yes | depends on policy |

## Git Policy for Generated Context

Generated context files fall into two categories:

### Committed Generated Context

Context artifacts that are committed to git and reviewed:

```text
.monad/context/current-state.md
.monad/context/latest-handoff.md
.monad/context/latest-context-pack.md
.monad/context/decision-log.md
.monad/context/session-chronicles/
```

These are generated, reviewed by a human, and committed. They become part of the repository record.

### Ephemeral Generated Context

Context artifacts that are generated on demand and excluded from git:

```text
.monad/context/generated/repository-context-pack.md
.monad/context/generated/repository-context-pack.json
```

These are created by `monad context --write` and ignored via `.gitignore`. They help the current session but are not part of the repository record.

## Determinism

Generated artifacts should be deterministic:

- The same repository state should produce the same output.
- File listings should be sorted.
- Sections should appear in a fixed order.
- Timestamps should reflect generation time, not random values.

## Source File Tracking

Every generated artifact must include a `source_files` field in frontmatter or a "Source Files Used" section listing all files that were read to produce the artifact.

This makes generated context auditable. A reviewer can check whether the generator used the right inputs.

## Staleness Detection

Generated artifacts include enough metadata for staleness detection:

- `generated_at` date;
- `epic` and `work_packet` at generation time;
- `source_files` used.

A generated artifact is stale when the current epic or work packet differs from the values recorded at generation time.

## Validation

A generated artifact is valid when:

1. All required frontmatter fields are present.
2. All required sections for the artifact type are present.
3. No section is blank (use "None" or "No known blockers" explicitly).
4. The `source_files` field is populated.
5. The artifact type matches the expected schema from `CONTEXT-ARTIFACT-SCHEMAS.md`.
